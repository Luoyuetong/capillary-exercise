# 什么是 CI：从"单机开发"到"工程纪律"的跃迁

> 教学材料 · 测试与自动化篇（承接 002 写测试、003 测试工具）

前面讲了**为什么写测试**（002）和**用什么写测试**（003）。但还有一个关键问题没回答：

> **测试写好了，谁来保证它每次都被运行？**

这一篇讲 **CI（持续集成）**——它是什么、做了什么、为什么要做。更重要的是，它代表了一个本质的转变：**从"单机开发的个人手艺"，跃迁到"团队协作的工程纪律"。**

---

## 一、CI 是什么

**CI（Continuous Integration，持续集成）**：每次代码提交或合并请求时，**自动**在一个干净的环境里**编译 + 测试**，确认这次改动没把项目弄坏。

我们项目的 CI 配置（`.github/workflows/build-test.yml`）做的事：

```
你 push 代码 / 提 PR
      ↓ 自动触发
GitHub 开一台干净的云端机器
      ↓
① 拉取代码  ② 装 .NET  ③ 还原依赖  ④ 编译  ⑤ 跑测试
      ↓
全过 ✓ → 允许合并        有失败 ✗ → 挡住，必须修好
```

关键词是**"自动"**和**"干净环境"**——不靠人记得去做，每次都在一台与你本地无关的全新机器上验证。

---

## 二、CI 做了什么：一道不知疲倦的质量关卡

CI 的本质，是一道**自动的、对所有人一视同仁的质量关卡**。

它解决的是一个很现实的人性问题——**人会忘、会偷懒、会自我感觉良好**：

| 没有 CI（靠人） | 有 CI（靠系统） |
|---------------|---------------|
| "我本地能编译就行了吧" | 干净云端环境编译，戳破"在我机器上能跑"的假象 |
| "测试等会儿再跑……"（然后忘了） | 每次提交**强制自动跑**，不依赖记忆 |
| 改 A 功能弄坏了 B，没发现 | 测试立刻报红，当场暴露 |
| 合并后才发现是坏的 | **CI 不过，PR 合不进去**，挡在门外 |
| 项目负责人"特权"绕过检查 | 规则对所有人平等，包括你自己 |

> 它不知疲倦、不讲情面、不会忘事——这正是"自动化"相对"人工自觉"的根本优势。

---

## 三、为什么要做：三个层次的理由

### 理由一：它超越了版本管理（和 Git 的本质区别）

很多人以为"用了 Git 就够了"。但 Git 和 CI 做的是两件不同的事：

| | Git / 版本管理 | CI |
|---|--------------|-----|
| 本质 | **记录**你做了什么 | **检查**你做得对不对 |
| 性质 | 被动存档 | 主动验证 |
| 回答 | "改了什么？谁改的？" | "这次改动，项目还正常吗？" |

> Git 是"历史账本"，CI 是"质检员"。账本只记录，不判断对错；质检员每次都把关。
> **从"记录"到"验证"，这是一个本质的跨越。**

### 理由二：它让"自动测试"真正有了价值

这是 CI 和测试主��（002、003）的关键连接：

**测试如果靠人记得去跑，迟早会烂掉。**

现实中常见的衰败过程：
```
项目初期：勤快地写测试、跑测试
     ↓ 赶进度
"这次改动小，先不跑测试了"
     ↓ 几次之后
测试开始报错，但没人管（因为没人跑）
     ↓ 最终
测试全烂，形同虚设 → 等于没写
```

**CI 斩断了这个衰败链条**：每次提交都自动跑测试，测试**不可能被遗忘、被跳过**。坏了立刻红灯，逼你当场修。

> 写测试（002）是"造了安全网"，CI 是"确保安全网每次都张开"。**没有 CI，再好的测试也会慢慢腐烂。**

### 理由三：它是从"单机开发"到"工程纪律"的跃迁

这是最深的一层。**单机开发**和**工程化开发**，在质量保障上有本质差别：

| | 单机开发（个人手艺） | 工程化开发（工程纪律） |
|---|---------------------|----------------------|
| 质量靠什么 | 个人自觉 + 记忆 + 手感 | 系统强制 + 自动验证 |
| 验证环境 | "我的电脑"（带着各种私货） | 干净标准环境 |
| 出错发现 | 靠运气、靠用户反馈 | 提交即检查，当场发现 |
| 能否偷懒 | 能（没人管） | 不能（流程挡着） |
| 可持续性 | 依赖个人状态，易松懈 | 制度化，稳定 |

单机开发的质量，系于"开发者今天累不累、记不记得、严不严格"——**飘忽不定**。
工程化开发，把质量保障**从个人品质中剥离出来，交给系统**——**稳定可靠**。

> 这就是"工程"二字的含义：不依赖某个人的英雄主义，而靠流程和工具，让普通人也能稳定产出可靠的结果。
> **CI，是这种工程纪律最具体的体现之一。**

## 四、和我们流程的关系

回到培训反复强调的"流程倒逼习惯"——CI 是其中关键一环：

```
你提 PR → CI 自动编译 + 测试
        ├─ 通过 ✓ → 才允许合并
        └─ 失败 ✗ → 挡住，必须修好再来
```

它把"每次改动都要保证能编译、测试能过"，从**"靠自觉"**变成了**"系统强制"**。这正是流程倒逼习惯的典型：你不想跑测试都不行，因为流程不放你过。

（呼应 000 总纲：好习惯靠制度保障，而非靠意志力硬扛。）

## 五、一个诚实的说明：我们 CI 的边界

我们项目的 CI 没有跑**全部**测试，这是有意的，也反映了真实工程中的权衡。

因为 GitHub 的云端机器上**没有 Jet OLEDB 驱动**（老的 32 位 Windows 组件），也不方便启动我们的 Mock 程序，所以：

- **单元测试**（用 NSubstitute 测业务逻辑，无外部依赖）→ **CI 跑**
- **数据库测试 / 集成测试 / 端到端测试**（需真实驱动或 Mock 程序）→ CI 跳过，**本地跑**

CI 的定位因此很清晰：**守住"代码能编译 + 核心业务逻辑正确"这条最低门槛**。它不求测全，但守住了最关键的底线。

> 这也是一个工程判断：CI 该跑什么、不该跑什么，取决于成本和价值的权衡，不是"越多越好"。

## 六、最后：聊聊"门槛"

和这个系列的其他材料一样，得说一句"门槛"。

过去，搭一套 CI 是**重活**——要自己维护构建服务器（如 Jenkins）、配置环境、保证它一直在线。成本高到很多小团队、个人开发者直接放弃，继续"单机开发"，这**情有可原**。

**但今天，门槛大大降低了。**

像 GitHub Actions 这样的工具，让 CI 变成了**一个 YAML 配置文件**的事——几十行，提交上去就自动生效，云端机器免费用。我们项目的整个 CI，就是 `build-test.yml` 这一个文件。

> 过去：搭 CI 成本高 → 单机开发情有可原
> 现在：一个配置文件搞定 → 没理由不享受工程化的保障

门槛降下来了，"工程纪律"不再是大团队的专利——**哪怕你一个人做项目，也能拥有一道自动质量关卡。**

---

## 一句话总结

> **CI 是一道每次提交都自动运行的"编译 + 测试"质量关卡。**
> **它超越了版本管理（从记录到验证），让自动测试不被遗忘（从造安全网到确保张开），更标志着开发从"靠个人自觉的单机手艺"，跃迁到"靠系统保障的工程纪律"——而今天，这道关卡只需一个配置文件就能拥有。**
