# 全自动劈刀发料机 Demo - 程序设计文档

## 一、项目概况

- **设备名称**：全自动劈刀发料机
- **功能**：劈刀(Capillary)自动存入机台，根据随件单领料信息自动领料
- **仓位**：A/B/C 三面，每面18×18矩阵（共972个仓位）
- **坐标定义**：AX?Y?、BX?Y?、CX?Y?

##

| 项目 | 选择 | 说明 |
|------|------|------|
| 开发语言 | C# | .NET 6+ |
| UI框架 | WinForms | 桌面应用 |
| 数据库 | Access / OleDb | 与参考代码保持一致（Jet OLEDB 4.0） |
| PLC通信 | 独立Mock程序（MockPLC） | 简化TCP协议，教学环境无PLC硬件 |
| 扫码器 | 独立Mock程序（MockScanner） | TCP通信，模拟真实独立设备架构 |
| MES/Oracle | 独立Mock程序（MockMES） | HTTP API，教学环境无MES/Oracle服务器 |

## 三、与参考代码的对照改进

参考代码（ZunY_PD_AutoSR）的问题及本项目的改进策略：

| 参考代码问题 | 本项目改进 |
|-------------|-----------|
| MainClass 全局静态类承担过多职责 | 按职责拆分为独立Service，通过构造函数注入依赖 |
| UI与业务逻辑耦合（逻辑写在按钮事件中） | 业务流程封装在Service层，Form只负责UI交互 |
| Form之间通过静态变量传递数据 | 流程上下文通过方法参数和返回值传递 |
| SQL字符串拼接（注入风险） | 使用参数化查询（同样使用Access/OleDb） |
| PLC寄存器地址硬编码散布各处 | 集中定义为常量（本Demo使用Mock，但保留该结构） |
| 命名不规范、命名空间混乱 | 统一命名空间 `CapillaryDemo`，英文命名 |
| 无接口抽象，硬件调用直接写死 | 所有硬件交互通过接口抽象，支持Mock/真实切换 |
| 忙等待自旋锁同步PLC线程 | 使用 `async/await` + `SemaphoreSlim` |

## 六、硬件接口设计

### 6.1 IScanner（扫码器接口）

```csharp
public interface IScanner
{
    /// 连接扫码器
    Task<bool> ConnectAsync();

    /// 断开连接
    void Disconnect();

    /// 触发一次扫码，返回条码；失败返回null
    Task<string?> ScanAsync(CancellationToken ct = default);

    /// 连接状态
    bool IsConnected { get; }

    /// 收到扫码结果事件（被动推送模式）
    event Action<string> OnBarcodeReceived;
}
```

**TcpScannerClient实现**：连接MockScanner的TCP端口（127.0.0.1:9001），支持两种模式：
- **主动请求**：发送 `SCAN\n` 命令，等待返回条码
- **被动监听**：MockScanner主动推送条码，触发 `OnBarcodeReceived` 事件

### 6.2 IPlcController（PLC控制接口）

```csharp
public interface IPlcController
{
    Task<bool> ConnectAsync();
    void Disconnect();
    bool IsConnected { get; }

    Task<bool> GrabToScanPositionAsync();
    Task<bool> StoreToSlotAsync(string face, int x, int y);
    Task<bool> DropToErrorBinAsync();
    Task<bool> FetchFromSlotAsync(string face, int x, int y);
    Task<bool> OutputToPickupPortAsync();
    Task<bool> ReturnToSlotAsync(string face, int x, int y);
    Task<bool> OpenReturnPortAsync();
    Task<bool> CloseReturnPortAsync();
    Task<bool> DetectMaterialInReturnPortAsync();

    event Action<string> OnStatusChanged;
}
```

**TcpPlcClient实现**：连接MockPLC的TCP端口（127.0.0.1:9002），将高层动作指令转为简化的寄存器读写命令发送给MockPLC。生产环境替换为基于HslCommunication的FINS协议实现。

### 6.3 IMesService（MES接口）

```csharp
public interface IMesService
{
    Task<string?> QueryCapillaryTypeAsync(string workOrder, string machineNo);
    Task<bool> ReportPickupAsync(string workOrder, string machineNo,
                                  string barcode, string capillaryType);
    Task<bool> ValidateReturnAsync(string barcode, string machineNo);
}
```

**HttpMesClient实现**：通过HTTP调用MockMES的API端点。生产环境只需将URL改为真实MES服务器地址。

## 八、核心流程

### 8.1 存料流程

```
启动补料
  │
  ▼
PLC: 抓取物料至读码位 ── GrabToScanPositionAsync()
  │
  ▼
扫码器: 读码 ── ScanAsync()
  │
  ├─ 失败 → PLC丢入异常仓 → 记录日志 → 结束
  │
  ▼ 成功
系统: 校验条码（格式、是否重复入库）
  │
  ├─ 失败 → PLC丢入异常仓 → 记录日志 → 结束
  │
  ▼ 成功
SlotManager: 分配空仓位
  │
  ▼
PLC: 存入仓位 ── StoreToSlotAsync(face, x, y)
  │
  ▼
DB: 写入 CapillaryInfo + OperationLog
  │
  ▼
完成
```

### 8.2 领料流程

```
输入工单号 + 机台号
  │
  ▼
MES: 查询劈刀类型 ── QueryCapillaryTypeAsync()
  │
  ▼
SlotManager: FIFO查找最早入库的该类型劈刀
  │
  ├─ 无库存 → 报警提示 → 结束
  │
  ▼
PLC: 取出劈刀至读码位 ── FetchFromSlotAsync()
  │
  ▼
扫码器: 读码核对
  │
  ├─ 失败 → PLC放回原位 → 锁定仓位 → 结束
  │
  ▼ 成功
MES: 上报领料 ── ReportPickupAsync()
  │
  ├─ MES拒绝 → PLC放回原位 → 锁定仓位 → 结束
  │
  ▼ 确认OK
PLC: 出料 ── OutputToPickupPortAsync()
  │
  ▼
DB: 更新状态=已领出 + 记录日志
  │
  ▼
完成
```

### 8.3 退料流程

```
扫入劈刀条码 + 输入机台号
  │
  ▼
DB: 校验（条码是否已领出、机台号是否匹配）
  │
  ├─ 失败 → 提示原因 → 结束
  │
  ▼ 成功
PLC: 打开退料口 ── OpenReturnPortAsync()
  │
  ▼
PLC: 等待检测物料放入 ── DetectMaterialInReturnPortAsync()
  │
  ▼
PLC: 关闭退料口 ── CloseReturnPortAsync()
  │
  ▼
DB: 更新状态 + 记录日志
  │
  ▼
完成
```

## 九、Mock独立程序设计

三个Mock程序均为独立WinForms应用，各自提供网络服务端，主程序通过网络连接它们。

### 9.1 MockScanner（模拟扫码器）

**通信协议**：TCP，监听 `127.0.0.1:9001`，UTF-8文本，`\n` 分隔

| 方向 | 命令 | 说明 |
|------|------|------|
| 主程序 → MockScanner | `SCAN\n` | 触发扫码请求 |
| MockScanner → 主程序 | `OK:{barcode}\n` | 扫码成功 |
| MockScanner → 主程序 | `FAIL\n` | 扫码失败 |

**界面**：条码输入框、「发送」「模拟失败」「随机生成」按钮、连接状态、通信日志

### 9.2 MockPLC（模拟PLC控制器）

**通信协议**：TCP，监听 `127.0.0.1:9002`，UTF-8文本，`\n` 分隔

| 方向 | 命令 | 说明 |
|------|------|------|
| 主程序 → MockPLC | `CMD:GRAB_TO_SCAN\n` | 抓取至读码位 |
| 主程序 → MockPLC | `CMD:STORE:{face},{x},{y}\n` | 存入指定仓位 |
| 主程序 → MockPLC | `CMD:FETCH:{face},{x},{y}\n` | 从仓位取出 |
| 主程序 → MockPLC | `CMD:DROP_ERROR\n` | 丢入异常仓 |
| 主程序 → MockPLC | `CMD:OUTPUT\n` | 放入出料口 |
| 主程序 → MockPLC | `CMD:RETURN:{face},{x},{y}\n` | 放回原仓位 |
| 主程序 → MockPLC | `CMD:OPEN_RETURN\n` | 打开退料口 |
| 主程序 → MockPLC | `CMD:CLOSE_RETURN\n` | 关闭退料口 |
| 主程序 → MockPLC | `CMD:DETECT_MATERIAL\n` | 检测退料口物料 |
| MockPLC → 主程序 | `OK\n` | 动作执行成功 |
| MockPLC → 主程序 | `FAIL:{reason}\n` | 动作执行失败 |

**界面**：当前动作状态显示、动作耗时设置（模拟快/慢）、「模拟故障」开关、动作执行日志

### 9.3 MockMES（模拟MES服务器）

**通信协议**：HTTP，监听 `http://127.0.0.1:9003`

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/capillary/query` | POST | 根据工单+机台号查询劈刀类型 |
| `/api/capillary/pickup` | POST | 上报领料信息 |
| `/api/capillary/return` | POST | 验证退料信息 |

请求/响应均为JSON格式，与ref code中MTMS API风格一致。

**界面**：预置测试数据管理（工单→劈刀类型映射表）、API调用日志、「模拟拒绝」开关

## 十二、生产环境切换指南

从教学Demo切换到生产环境，**主程序业务逻辑代码无需修改**，只需以下调整：

### 12.1 PLC：MockPLC → Omron真实PLC

### 12.2 扫码器：MockScanner → 真实扫码器
