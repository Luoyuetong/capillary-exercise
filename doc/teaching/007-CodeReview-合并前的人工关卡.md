# Code Review：合并前的最后一道人工关卡

> 教学材料 · 流程与质量篇（承接 005 CI）

代码写完、测试通过、CI 变绿——可以合并了吗？

**还差一步：Code Review（代码审查）。**

CI 守住了"能编译、测试过、格式对"这条底线，但有一类问题机器永远看不出来——设计好不好、逻辑清不清、半年后还看得懂吗。这些，要靠人在合并前再看一眼。

这一篇讲：Code Review 审什么、为什么单人项目也要做、它和 CI 怎么分工，最后讲——**引入 AI 后，审查这件事有哪些改变。**

---

## 引子：为什么 CI 绿了还不够

先厘清一个常见误解：

> "CI 都通过了，代码肯定没问题，直接合并不就行了？"

不行。CI 能证明的，只是**底线**：
- 代码能编译 ✓
- 测试能通过 ✓
- 格式符合规范 ✓

但它证明不了**上限**：
- 这个设计合理吗？有没有更简单的做法？
- 业务逻辑有没有混进 UI 层？
- 这段代码三个月后别人（或你自己）读得懂吗？
- 有没有偏离这个 Issue 最初的目标？

**这些是"质量"问题，不是"对错"问题，机器判断不了，必须靠人。** 这就是 Code Review 存在的理由。

---

## 一、Code Review 审什么

回顾 005 讲的规则分类——Code Review 专管机器管不了的那两类：

| 审查维度 | 具体看什么 | 举例 |
|---------|-----------|------|
| **设计** | 结构是否合理、职责是否清晰 | 是否面向接口？依赖是否注入？ |
| **分层** | 各层职责有没有越界 | 业务逻辑有没有写进按���事件？ |
| **约定** | 是否符合设计决策 | 硬件失败是返回值还是抛异常？（见设计 5.2） |
| **可维护性** | 可读性、命名意图、注释质量 | 半年后能看懂吗？注释讲清"为什么"了吗？ |
| **完整性** | 是否真的实现了 Issue 要求 | 异常分支漏了吗？测试覆盖够吗？ |

**注意：不审什么——格式、缩进、命名大小写。** 那些是 CI 和工具的活，人不该浪费精力盯（见 005 的人机分工）。

> Code Review 的精力，要花在"机器看不出、但影响长期质量"的地方。

## 二、单人项目，自己审自己的代码有意义吗？

这是最容易被质疑的点：**"就我一个人，自己审自己写的，能审出什么？"**

有意义，而且很重要。关键是**制造"视角的距离"**——让"审查者的你"和"开发者的你"不是同一个状态。三个办法：

### 办法一：拉开时间差
写完别马上审。隔几小时、甚至过一夜再回来看。

人刚写完代码时，脑子还沉浸在"我是怎么实现的"，看什么都觉得对。**隔一段时间，实现细节淡忘了，你才能以接近"陌生人"的眼光重新读它**——很多当时觉得理所当然的地方，这时会突然觉得"这写的什么？"

### 办法二：先读自己的 PR 描述
提 PR 时强迫自己写清楚"做了什么、为什么这么做"。

**如果你发现自己写不清楚、解释不顺——那往往说明代码本身就没想清楚。** PR 描述是一面镜子，照出你思路里的含糊。

### 办法三：对照原始 Issue
审查时把 Issue 拉出来，逐条对：
- Issue 要求的，都实现了吗？
- 验收标准，都满足了吗？
- 有没有写着写着，偏离了最初的目标，或者顺手加了不该加的东西？

> 这三个办法的本质，都是教学材料 000 第四条讲的——**用流程逼一个人切换角色，从不同角度看同一份代码**。审查时的你，要刻意扮演一个"挑剔的陌生人"，而不是"护短的作者"。

## 三、Code Review 在流程里的位置

它在**迭代开发循环**中，PR 提交后、合并前：

```
对每个 Issue：
  建分支 → 编码（遵守规范）→ 本地跑测试 → 提 PR
                                            ↓
                              CI 自动验证（编译/测试/格式）
                                            ↓
                          【Code Review：人工审设计与质量】 ← 在这里
                                            ↓
                              通过 → 合并 → 关闭 Issue
```

### 和 CI 的分工：守底线 vs 提上限

| | CI | Code Review |
|---|-----|------------|
| 执行者 | 机器 | 人 |
| 管什么 | 编译、测试、格式 | 设计、逻辑、可维护性 |
| 性质 | **守底线**（不达标就挡住） | **提上限**（让代码更好） |
| 时机 | 提交即自动跑 | CI 通过后，合并前 |

**两者互补，缺一不可**：CI 防止"坏代码"混进来，Code Review 防止"能跑但烂的代码"沉淀下来。

## 四、引入 AI 之后，审查有哪些改变

前面讲的是 AI 之前就成熟的做法。现在加入 AI——**它作为一种新工具，恰好补上了单人自审最大的短板。**

### 单人自审的天然短板
再怎么拉开时间差、切换角色，**终究是同一个脑子**。你的知识盲区、思维定式，换个角色也还在——你不知道的问题，自己审一百遍也发现不了。这是单人项目绕不过的局限。

### AI 当"第二双眼睛"
把代码 diff 交给 AI 审查，它能：
- 指出你没注意的潜在问题（"这里没做空值检查""这个分支没对应测试"）
- 从它见过的大量代码经验里，给出改进建议
- 帮你对照 Issue，查有没有遗漏

**这相当于给单人项目，凭空补上了一个"审查者"**——弥补了"同一个脑子"的盲区。

### 但责任边界要清楚
**AI 补的是"视角"，不是"责任"。**
- AI 会误报，也会因为不懂业务上下文而提错建议
- 最终判断"这个意见对不对、采不采纳"，依然是你的责任
- 你是决策者，AI 是参谋——这和整个系列反复强调的"人主导、AI 协助"一致

> AI 让单人也能享受"被审查"的好处，但它不替你拍板。看完 AI 的意见，判断和取舍仍然是人的事。

---

## 五、最后：聊聊"门槛"

和这个系列一贯的视角一样，说一句门槛。

过去，**单人项目几乎做不了真正的 Code Review**——没有第二个人，自审又受限于"同一个脑子"，所以很多人干脆跳过，写完就合。**这情有可原**：没人可审，是客观条件限制。

**但今天，门槛降低了。**
- AI 可以随时当"第二双眼睛"，帮你审 diff、挑问题、查遗漏
- 单人项目第一次能低成本地获得"被审查"的价值

> 过去：单人没人可审 → 跳过 Code Review 情有可原
> 现在：AI 当第二双眼睛 → 单人也能做有效的代码审查

哪怕你一个人做项目，也能拥有一道像样的"合并前人工关卡"——只不过这道关卡，现在由"你 + AI"一起把守。

---

## 一句话总结

> **CI 守底线（能不能跑），Code Review 提上限（好不好）——前者靠机器，后者靠人，缺一不可。**
> **单人自审的关键是切换角色、制造视角距离；而 AI 作为"第二双眼睛"，恰好补上了"同一个脑子"的盲区——但判断和拍板，始终是你的事。**
