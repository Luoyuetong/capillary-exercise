# Issue 与模块：为什么要提前规划 Issue

> 教学材料 · 软件工程流程与开发习惯

本文讲清两个问题：
1. **Issue 和模块到底有什么区别？**
2. **为什么建议提前规划 Issue？**（关键不只是工作安排，而是倒逼全局思考）

---

## 一、Issue 与模块的区别

很多人把 Issue 和模块混为一谈，觉得"一个 Issue 就是写一个类/一个文件"。其实它们是**两个不同维度**的概念。

### 1.1 一句话区分

> **模块是"代码怎么组织"，Issue 是"工作怎么交付"。**

### 1.2 详细对比

| 维度 | 模块（Module） | Issue |
|------|---------------|-------|
| **本质** | 代码组织单位 | 工作任务单位 |
| **关注点** | 技术结构、复用、解耦 | 可交付价值、进度、验收 |
| **典型例子** | `PickupService` 类、`IPlcController` 接口 | "实现领料业务逻辑"、"定义硬件接口" |
| **粒度依据** | 技术边界（单一职责） | 工作边界（可独立完成和验收） |
| **测试** | 单元测试（验证逻辑正确） | 验收测试（验证任务完成） |
| **独立性** | 可以依赖其他模块 | 尽量减少依赖，便于并行开发 |

### 1.3 关键澄清：一个常见误解

❌ **误解**："模块不一定能独立测试，Issue 才能独立测试"

✅ **更准确**：
- **模块应该能独立测试**——这正是单元测试的基础（通过 Mock 隔离依赖）
- **真正的区别在于"交付"**：
  - 模块能独立**测试**，但不一定能独立**交付价值**
  - Issue 必须能独立**交付和验收**——做完它，能证明"这个任务完成了"

### 1.4 用本项目举例

**模块视角**（代码怎么组织）：
- `IPlcController` 是一个模块（接口）
- `TcpPlcClient` 是一个模块（实现类）
- `PickupService` 是一个模块（业务逻辑类）

**Issue 视角**（工作怎么交付）：

| Issue | 包含的模块 | 交付物 |
|-------|-----------|--------|
| #3 定义硬件接口 | `IPlcController` + `IScanner` 两个模块 | 接口定义（不含实现） |
| #4 实现 TcpPlcClient | `TcpPlcClient` 一个模块 | 能连 MockPLC 并测试通过 |
| #7 实现 PickupService | `PickupService` 一个模块 | 完整流程 + 单元测试 |

可以看到：
- 一个 Issue 可以包含**多个模块**（如 #3）
- 一个 Issue 也可以只是某个功能的**一部分**（如 #3 只定义接口，#4 才实现）
- **Issue 的边界是"能不能独立交付验收"，不是"是不是一个类"**

### 1.5 拆 Issue 和划分模块时，分别在想什么？

**拆 Issue 时问自己**：
1. 这个任务做完，能交付什么？（代码 + 测试 + 文档）
2. 验收标准是什么？（怎么证明做完了）
3. 依赖关系如何？（能不能和别的任务并行）
4. 粒度合适吗？（太大无法并行，太小频繁切换）

**划分模块时问自己**：
1. 职责单一吗？（单一职责原则）
2. 耦合度低吗？（高内聚、低耦合）
3. 可测试吗？（依赖能 Mock 掉）
4. 可复用吗？（未来能否在别处用）

## 二、为什么建议提前规划 Issue

提前规划 Issue 有三层价值，**一层比一层重要**。

### 2.1 第一层：便于工作安排（看得见的好处）

- **任务清单化**：把"做一个领料系统"这种模糊目标，拆成 13 个具体任务
- **进度可视化**：哪些做完了、哪些在做、哪些没开始，一目了然
- **并行与排期**：理清依赖关系后，知道什么能同时做、什么必须先做

这一层是大家都能想到的，但**不是最重要的**。

### 2.2 第二层：便于测试（容易忽略的好处）

- **每个 Issue 都有验收标准**——写 Issue 时就被迫想"做完怎么验证"
- **测试驱动**：先想清楚"怎么证明它对"，再写实现
- **Mock 提前**：本项目中，正是因为规划 Issue 时发现"客户端要测试就得先有 Mock"，才把 Mock 程序（#11-#13）提前

这一层比第一层深，但**仍然不是最关键的**。

### 2.3 第三层：倒逼全局思考（最重要的价值）⭐

**这才是提前规划 Issue 的核心意义。**

写 Issue 的过程，本质上是**逼自己把整个系统在脑子里过一遍**：

> 要拆出合理的 Issue，你必须先回答：
> - 这个系统由哪些部分组成？
> - 它们之间怎么依赖？谁先谁后？
> - 每个部分的边界在哪？做到什么程度算完成？
> - 哪些地方有风险、有不确定性？

**换句话说：你没法在不理解全局的情况下，拆出合理的 Issue。**

#### 对比两种开发方式

**❌ 上手就干（没有全局观）**：
```
拿到需求 → 直接打开编辑器写代码 → 写到哪算哪
         → 写到一半发现漏了 MES 接口 → 临时加
         → 又发现没法测试 → 临时补 Mock
         → 最后代码东拼西凑，自己都理不清结构
```

**✅ 先规划 Issue（建立全局观）**：
```
拿到需求 → 写设计文档（理清架构）
         → 拆 Issue（强迫自己想清楚每个部分和依赖）
         → 发现"客户端要测试得先有 Mock" → 调整顺序
         → 发现"接口定义和实现可以分开" → 拆成两个 Issue
         → 动手时，心中有完整的图
```

**关键差异**：第二种方式，在**写一行代码之前**，就已经把系统想明白了。

#### 这就是"流程倒逼习惯"

很多人（包括有经验的开发者）有"上手就干"的惯性。**强制要求"先拆 Issue 才能开发"**，就是用流程倒逼你：

> 不想清楚全局 → 拆不出合理的 Issue → 流程过不去 → 被迫先思考

这不是形式主义，而是**用一个看得见的动作（拆 Issue），逼出一个看不见的能力（全局思考）**。

### 2.4 三层价值小结

| 层次 | 价值 | 重要性 |
|------|------|--------|
| 第一层 | 便于工作安排（任务清单、进度、排期） | ⭐ |
| 第二层 | 便于测试（验收标准、测试驱动、Mock 提前） | ⭐⭐ |
| 第三层 | **倒逼全局思考（先想清楚再动手）** | ⭐⭐⭐ |

**记住**：提前规划 Issue，表面是管理工具，本质是**思考工具**。

## 三、本项目的真实案例

在 `capillary-exercise` 项目中，提前规划 Issue 的价值体现得很具体：

### 案例 1：拆 Issue 时发现 Mock 顺序问题
- **初始拆解**：把所有 Mock 程序放在最后的"集成测试"Issue（#9）里
- **拆解中反思**：客户端（#4-#6）开发时就需要 Mock 来测试，不能等到最后
- **调整**：把 Mock 独立成 #11-#13，提前到客户端之前
- **价值**：这个问题是在**写 Issue 时**发现的，而不是写代码写到一半才发现

### 案例 2：拆 Issue 时理清了依赖
- 写 Issue 的过程，强迫梳理出 6 个开发批次的依赖顺序
- 哪些能并行（#1、#2、#3）、哪些必须先做（Mock 先于客户端）
- **价值**：动手前就知道了整个开发路径

### 案例 3：接口与实现分离
- 拆 Issue 时意识到："定义接口"和"实现接口"是两件事
- 拆成 #3（定义 IPlcController/IScanner）和 #4/#5（实现）
- **价值**：体现了"面向接口设计"的思路，也让任务更聚焦

> **演进说明（2026-06-22）**：本节案例记录的是**初次拆 Issue 时**的真实思考，忠实保留。进入迭代开发后，为聚焦工程流程教学，独立 Mock 程序（#11-#13）+ TCP 客户端（#4-#6）改为 3 个进程内 Fake（合并为 #4），数据库改 SQLite。案例 1 "Mock 要提前"的洞察在新方案下依旧成立——只是"提前准备的可测试替身"从独立 Mock 程序变成了进程内 Fake。详见 `000-PROJECT_STARTUP.md` 附录与 `003-ISSUE_LIST.md` 开头的设计演进说明。

## 四、给学员的建议

1. **别跳过拆 Issue 这一步**——哪怕是单人项目，哪怕觉得"我心里有数"
2. **拆 Issue 时多问几个"为什么"**——为什么这样拆？依赖对不对？能测试吗？
3. **允许 Issue 粗糙，但不允许没有**——想不透没关系，先有框架，边做边调
4. **把"拆不出合理 Issue"当成信号**——如果拆不出来，说明你对系统的理解还不够，正好回头补设计

---

## 五、一句话总结

> **模块是代码的组织方式，Issue 是工作的交付单元。**
> **提前规划 Issue，表面是为了管理和测试，本质是逼自己在动手前建立全局观。**
